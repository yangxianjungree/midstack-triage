# 过程 3: 想从底层文件找异常, 但仍然缺前置证据

## 定位

- **过程性质**: 失败的排障过程
- **当时起点**: 用户要求重新来一次, 标准改成 "有理有据, 没证据就承认没有"
- **这轮产出**: 进一步证明 "直接看底层文件" 不具备工程可行性
- **保真说明**: 这一轮保留下来的是思路转折和用户纠偏, 不是一组有结论力的现场命令输出

## 原始实际经过

### 步骤 1: 任务被重置为重新现场排查

拿到什么:

- 用户要求重新排查
- 用户明确说, 如果没有证据, 可以直接承认没有证据

当时怎么想:

- 这意味着排障标准已经非常清楚: 不能猜, 不能倒推, 只能靠证据

为什么执行下一步:

- 需要重新选择一个可以被证据支持的收敛路径

实际动作:

- 重新思考下一步应该看哪里

返回结果:

- 现场证据并没有比过程 1 多出根因级信息

### 步骤 2: 思路又试图落到底层文件

拿到什么:

- 仍然只有 Pod 启动失败和日志很少这类上层现象

当时怎么想:

- 试图通过数据目录, 数据文件, 底层字节异常之类的东西来缩小范围

为什么执行下一步:

- 这本质上还是在寻找一个捷径
- 但这个捷径并没有上游证据支撑

实际动作:

- 这一步没有形成新的有效取证链
- 对话里留下来的主要是思路方向, 不是强证据输出

返回结果:

- 没有新的根因级证据

这一小步的结论:

- 仍然只是猜测, 不是证据

### 步骤 3: 用户从工程可行性上否掉这条路

拿到什么:

- 用户追问 "为什么会想到去对比数据目录下的数据文件, 出发点是什么"
- 用户给出更严格的现场约束, 比如上百个表, 10T 数据量
- 用户指出, 现场工程师不可能靠肉眼看文件来排查
- 用户还指出, 就算某个地方看到 `fffff`, 也不能自动推出 "数据一定异常"

当时怎么想:

- 这些质疑击中了这条路的根本问题
- 如果没有日志或错误码先把范围缩到文件层, 看文件这件事既不经济, 也不通用

为什么这轮必须停下:

- 因为再往下走, 只会继续消耗精力在没有正当性的方向上

实际动作:

- 停止把 "看底层文件" 当成默认建议

返回结果:

- 得到的是一条方法论上的否定结论

## 这轮为什么不成立

- 缺少把范围缩到 "本地存储引擎文件层" 的前置证据
- 缺少可以规模化复用的工程可行性
- 缺少从观测面到根因面的中间桥梁

## 反模式示例

下面这些动作不是说永远不能做, 而是说在这轮所掌握的证据条件下, 不应该成为默认起手式:

```bash
kubectl exec -n psmdb-test bnmongo-shard1-data-2 -- ls -lh /bitnami/mongodb/data/db
kubectl exec -n psmdb-test bnmongo-shard1-data-2 -- find /bitnami/mongodb/data/db -maxdepth 2 -type f | head -n 20
kubectl exec -n psmdb-test bnmongo-shard1-data-2 -- hexdump -C /bitnami/mongodb/data/db/journal/WiredTigerLog.0000000007 | head
```

这些动作缺的不是执行能力, 而是执行正当性.

## 这轮给后续工程师的真正提示

如果你发现自己已经开始想:

- "先去数据目录里找点异常再说"
- "先看看哪个文件长得怪"
- "先看看底层字节是不是不对"

那就应该停一下, 问自己两个问题:

- 我现在手里的证据, 有没有先把范围缩到文件层
- 如果这套系统有上百个表和 10T 数据, 我还会不会推荐别人这样排

如果这两个问题的答案都是否, 那说明方向错了.
