# 排障流程健壮性与实用性背景

本文件记录围绕 Midstack Triage 三个 slash 命令和后台五段式排障流程的背景讨论。

本文件是过程性背景记录，不是 L1 事实源。当前事实源仍以 `docs/specs/`、`core/models/`、`core/templates/`、`core/taxonomies/` 为准。

## 当前命令模型理解

当前用户侧只暴露三个主命令：

- `/midstack:start`
  - 对应第 1、2 段。
  - 负责启动排障、冻结输入、确认远程和 Kubernetes 基础可用性、做轻量对象盘点。
  - 输出 incident 是否 `ready` 或 `blocked`。
- `/midstack:analyse`
  - 对应第 3、4、5 段。
  - 负责信号采集与治理、推理诊断与深入验证、结论整合与知识沉淀。
  - 这是用户拿到阶段性排障结论和报告的主路径。
- `/midstack:review`
  - 不属于用户排障主路径。
  - 主要用于评价插件本次排障表现、评分、记录偏差和改进建议。

命令是用户交互层，五段式流程是内部执行模型。`incident` 目录和 `adapter output` 是两者之间的稳定合同。

## 后台五段式流程

当前流程基线为：

1. 受理与启动
2. 环境确认与对象盘点
3. 信号采集与治理
4. 推理诊断与深入验证
5. 结论整合与知识沉淀

当前讨论的重点不是调整这五段，而是评估这套流程在真实排障中的健壮性和实用性。

## 关键背景：不要把流程做成一次性流水线

当前设计方向是合理的，但真实排障不应被简化成：

```text
一次性采集 -> 一次性推理 -> 一次性结论
```

更实用的形态应允许：

```text
初始采集 -> 假设 -> 发现证据缺口 -> 定向补采 -> 修正假设 -> 阶段性收敛
```

这不一定意味着第一版就要实现完整自动循环，但流程和记录结构应能表达这种排障方式，避免后续被固定流水线锁死。

## k8s-project 案例启发

可参考兄弟仓库案例：

- `/home/stephen/AI/k8s-project/docs/case-studies/2026-06-09-mongo-startup-journal-corruption/README.md`

案例背景：

- 环境：`@test`
- 命名空间：`psmdb-test`
- 故障对象：`bnmongo-shard1-data-2`
- 最终证实根因：WiredTiger journal 文件损坏，导致 `mongod` 启动失败

该案例将同一故障拆成 4 次排障过程：

- 过程 1：只用 Kubernetes 表层证据做首轮分层。
  - 可以把问题缩到 `mongod` 启动失败。
  - 不能推出 WiredTiger、journal、数据损坏等根因。
- 过程 2：已知答案后直接围绕 `journal/` 推演。
  - 暴露了 answer-led bias。
  - 这是从答案倒推现场，不是现场证据驱动的排障。
- 过程 3：试图从底层文件层面反推。
  - 缺少从观测面到文件层的证据桥梁。
  - 不适合作为默认起手路径。
- 过程 4：先解释为什么 `kubectl logs` 很少，再找到真实 MongoDB 日志源。
  - 通过配置确认 MongoDB 日志写文件。
  - 追到节点侧 Pod 卷里的真实 `mongodb.log`。
  - 最终由 MongoDB 自己的日志直接给出 journal corrupted 根因。

这个案例对 Midstack Triage 的启发不是“多采一点日志”，而是：

- 有效排障来自证据推动的问题重写。
- 失败路径常见于停在表层、答案倒推、跳层取证。
- 根因级结论必须由足够直接的证据支撑。
- 历史答案只能作为假设来源，不能替代当前现场证据。

## 对 `/midstack:start` 的启发

`start` 的健壮性重点不是提前诊断，而是建立可靠现场入口：

- 冻结最小输入和第一手原始线索。
- 明确当前环境是否可进入。
- 明确目标中间件和初步对象范围。
- 记录用户线索，但不把用户线索当成已证实结论。
- 把当前已知事实和待确认问题分开。

`start` 不应为了“显得聪明”而提前跳到根因判断。

## 对 `/midstack:analyse` 的启发

`analyse` 应是证据链形成的主战场。

它需要避免只做固定清单采集后直接写结论。更稳健的行为是：

- 先用第 3 段建立初始证据包。
- 第 4 段基于证据生成多条候选假设。
- 明确每条假设的支持证据、反证条件和证据缺口。
- 当缺口影响根因判断时，优先形成定向补采问题。
- 如果无法补采，应限制结论层级和置信度，而不是强行给根因。

例如在 k8s-project 案例中，`kubectl logs` 很少不应被当成普通留白，而应被提升为一个关键调查问题：

```text
MongoDB 的真实日志源在哪里？
```

这类问题重写能力，是从表层现象走向根因证据的关键。

## 对 `/midstack:review` 的启发

`review` 不应只评价最终文本是否完整，也应评价排障过程质量。

可评价的过程偏差包括：

- 是否把用户线索或历史答案当成现场证据。
- 是否出现 answer-led bias。
- 是否从表层信号直接跳到深层根因。
- 是否缺少从观测面到根因面的证据桥梁。
- 是否在证据不足时仍给出高置信根因。
- 是否明确记录了关键证据缺口和下一步补采动作。
- 是否保留了足够的命令、返回结果、artifact 引用，方便复盘。

这会让 `review` 成为插件改进闭环，而不是单纯的结果打分。

## 当前流程应强化的原则

### 1. 结论分层

应区分：

- 现象级结论：哪个对象异常，异常表现是什么。
- 影响级结论：影响哪个拓扑单元或服务能力。
- 机制级结论：异常通过什么机制导致影响。
- 根因级结论：为什么发生。

只拿到 Kubernetes 表层信号时，可以高置信说明 `mongod` 启动失败，但不应高置信说明 journal 损坏。

### 2. 证据缺口一等化

证据缺口不只是失败列表。

它应能表达：

- 缺什么证据。
- 为什么重要。
- 是否影响根因层判断。
- 是预期拿不到，还是关键证据缺失。
- 下一步是否应补采。

### 3. 历史案例作为假设，不作为证据

相似历史案例、用户事后答案、专家经验都可以帮助生成假设和验证路径，但不能替代当前现场证据。

### 4. `analyse` 成功不等于一定给根因

阶段性成功可以是：

- 已明确现象和影响范围。
- 已排除一批常见方向。
- 已说明根因级证据不足。
- 已给出下一步最有价值的只读补采动作。

这比强行给错根因更实用。

### 5. 真实日志源、时间线、对象拓扑和同伴视角应进入默认思维框架

从当前参考资料和 k8s-project 案例看，健壮排障至少需要持续关注：

- 真实日志源，而不是默认 `kubectl logs` 完整。
- 时间线，而不是孤立状态快照。
- 对象拓扑和影响半径，而不是单 Pod 局部视角。
- 同伴视角或业务侧视角，而不是只看故障对象自身。

## 与外部参考的一致性

当前方向与项目参考资料中的方法论一致：

- Google SRE：结构化问题报告、分层诊断、假设驱动、避免把相关性误判为因果。
- Azure SRE Agent：evidence gathering -> hypothesis generation -> validation -> conclusion。
- Datadog Timeline：时间线作为调查主轴。
- PagerDuty Probable Origin / Past Incidents：历史模式可帮助缩小范围，但不能替代当前证据。
- Amazon OpenSearch Investigation Agent：保留多假设并对假设执行验证或排除。

## 后续讨论点

逐项讨论见 [`discussions.md`](./discussions.md)：

- `analyse` 是否应显式支持一轮或多轮“定向补采”。
- `collection_report` 是否需要区分 `expected_gap` 与 `critical_gap`。
- `analysis.yaml` 是否需要更明确表达结论层级和置信度上限。
- `review` 是否需要加入过程偏差评分，而不只是结果质量评分。
- 历史案例和知识资产如何进入第 4 段：作为假设来源、验证路径来源，还是推荐参考资料。

## 当前暂不讨论

本背景记录暂不展开：

- 具体代码实现方式。
- 具体字段 schema 设计。
- remote executor 的内部错误分类。
- MongoDB 采集脚本的细节改造。

这些内容应在流程原则确认后，再进入单独提案或 L1 变更。
