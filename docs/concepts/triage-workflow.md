# Triage Workflow

本文件是排障主流程的概览（L2，解释性质）；规范性定义以[排障流程规范](../specs/triage-workflow.spec.md)为准。

可视化见 [Midstack 架构图](architecture-overview.md)（主图）及其中 [④ 推理验证展开](diagrams/supplements/architecture-phase4-detail.png)。

流程细节讨论见[排障流程讨论](../decisions/triage-workflow-discussion.md)。

## 结论

当前排障主流程采用 5 段式，作为后续 `runbook`、`skill`、表单和诊断工作流的基线：

1. `受理与启动`
2. `环境确认与对象盘点`
3. `信号采集与治理`
4. `推理诊断与深入验证`
5. `结论整合与知识沉淀`

## 1. 受理与启动

目标：

- 让排障能够启动
- 保存最小输入和原始线索

典型输入：

- 中间件类型
- 环境 IP
- 账号密码
- 客户提供的原始故障线索

## 2. 环境确认与对象盘点

目标：

- 确认目标环境和目标集群
- 识别部署架构
- 盘清相关排障对象

典型动作：

- 确认入口 IP 角色和跳转路径
- 确认本次报障对应哪一套集群
- 识别组件、实例、Pod、节点、Service、PVC 等对象

## 3. 信号采集与治理

目标：

- 收集并治理多源信号
- 将故障与具体对象建立关联
- 对信号进行去噪、归类、关联和范围收敛

典型动作：

- 收集状态、日志、监控、报错、变更等信号
- 确认哪些对象异常、哪些对象正常
- 关联信号与组件、实例、节点、服务等对象
- 尽量重建故障时间线
- 判断影响范围是单实例、单组件、单集群还是业务侧整体

## 4. 推理诊断与深入验证

目标：

- 复用历史经验
- 形成可验证的假设
- 通过深入验证持续收敛问题

典型动作：

- 检索历史故障记录和专家经验
- 生成多个候选假设
- 执行验证、排除错误假设、收敛最可信路径

## 5. 结论整合与知识沉淀

目标：

- 形成可交接、可复盘、可升级的阶段性结果
- 将有效经验回灌为后续可复用知识

输出内容通常包括：

- 当前现象
- 影响范围
- 关键证据
- 已验证结果
- 当前最可信结论
- 下一步建议
- 风险提示

知识沉淀通常包括：

- 新增或修订 runbook
- 补充历史案例
- 更新技能、命令或脚本
- 补充新的场景标签和经验索引

## 使用原则

- 先定主流程，再派生表单和字段
- 所有 `runbook` 和 `skill` 应尽量能映射到这 5 个阶段
- 不要求每次严格线性推进，但最终应能回归到这 5 段框架中
