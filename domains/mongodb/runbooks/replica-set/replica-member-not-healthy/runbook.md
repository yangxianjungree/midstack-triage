# MongoDB Replica Set Member Not Healthy

## 适用场景

当副本集成员出现以下表现时使用本 runbook：

- 成员状态不是预期的 `PRIMARY` 或 `SECONDARY`
- 某个成员长时间 `RECOVERING`、`STARTUP2` 或 `UNKNOWN`
- 复制延迟持续增大
- 业务侧怀疑读流量命中异常副本

## 目标

- 确认异常成员是谁
- 判断问题属于拓扑、网络、资源还是复制积压
- 在不执行高风险操作的前提下输出下一步建议

## Step 1: 确认副本集整体状态

执行：

```javascript
rs.status()
```

检查：

- 当前 `PRIMARY` 是否存在
- 各成员 `health` 是否正常
- 各成员 `stateStr` 是否符合预期
- 各成员 `optimeDate` 是否明显滞后

如果这里已经发现异常成员，记录成员名、状态和延迟情况，进入下一步。

## Step 2: 区分不可达和可达但异常

如果成员显示为 `UNKNOWN`、无心跳或连接失败，优先按“连接或网络问题”方向处理。

重点核验：

- Pod 或进程是否存活
- 目标端口是否监听
- 节点或容器网络是否异常
- 最近是否发生重建、迁移或网络策略变更

如果成员可达，但状态异常或复制明显滞后，继续下一步。

## Step 3: 检查复制积压和资源压力

重点核验：

- CPU 是否持续打满
- 磁盘 IO 是否饱和
- 可用磁盘空间是否不足
- 日志中是否存在复制错误、落后告警、存储层异常

判断逻辑：

- 如果资源异常明显，优先归类为资源问题导致的复制异常
- 如果资源正常但复制持续落后，优先归类为复制链路或数据积压问题

## Step 4: 输出结论

按以下格式输出：

```text
异常成员: <member>
当前状态: <state>
影响范围: <read-path/write-path/topology>
一级归因: <network|resource|replication|topology>
下一步建议: <next action>
```

## 安全说明

在本 runbook 范围内，不执行以下操作：

- `rs.stepDown()`
- `rs.reconfig()`
- 强制重启实例
- 强制重建副本

如果后续需要进入处置阶段，必须显式确认风险。
