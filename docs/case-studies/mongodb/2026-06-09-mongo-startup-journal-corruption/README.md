# `@test` Mongo 启动故障案例包

## 案例摘要

- **环境**: `@test`
- **命名空间**: `psmdb-test`
- **故障对象**: `bnmongo-shard1-data-2`
- **最终证实的根因**: WiredTiger journal 文件损坏, 导致 `mongod` 启动失败
- **案例目标**: 不是只给出答案, 而是把同一故障下 4 次排障路径拆开, 让后续工程师能对比 "为什么有的路径走不通, 为什么最后一条能拿到直接证据"

## 先读这个

本案例包刻意保留了 4 次过程的差异:

- 过程 1: 只做到 K8s 表层分层, 没拿到根因
- 过程 2: 已知答案后被带偏, 直接把 `journal/` 当起点
- 过程 3: 想从底层文件找异常, 但仍然缺前置证据
- 过程 4: 先找真实日志源, 再从 MongoDB 自己的日志拿到根因

其中 1 到 3 的价值不在 "答对了", 而在 "为什么这条路当时不成立". 过程 4 才是符合现场排障纪律的有效路径.

## 四次过程对比

| 过程 | 起点 | 主导问题 | 执行动作的依据 | 返回结果 | 结论性质 |
|------|------|----------|----------------|----------|----------|
| 1 | 只知道 Pod 起不来, `kubectl logs` 很少 | 先分层: 是 K8s 编排问题, 还是 `mongod` 启动问题 | K8s API 已经足够做第一层排除 | 只拿到 `CrashLoopBackOff`, `exitCode=14`, PVC/节点正常, 同伴视角 `Connection refused` | 只能缩到 `mongod` 启动失败, 还不是根因 |
| 2 | 用户事后披露了故障注入位置 | 已知答案会不会直接变成排障起点 | 注入命令触达 `journal/WiredTigerLog.0000000007` | 没有新增现场证据, 只有答案信息 | 这是反向推理, 不是现场排障 |
| 3 | 用户要求重新来, 但不给猜 | 能不能直接从底层文件反推问题 | 想走捷径, 试图从文件异常缩小范围 | 仍然没有前置证据把范围收窄到文件层 | 这条路不具备工程可行性 |
| 4 | 先解释为什么 `kubectl logs` 这么少 | 应用日志真正写到哪里 | 只有找到真实日志源, 才可能拿到根因 | 配置证实日志写文件, 节点侧文件日志直接报 `journal/WiredTigerLog.0000000007 corrupted` | 这是有效证据链 |

## 关于命令保真度

过程 4 保留了可复核命令和关键返回结果.

过程 1 到 3 没有在原始对话里保留完整 exact 命令串. 因此这些文件会明确区分:

- **原始实际经过**: 当时真的拿到了什么, 当时怎么想, 当时为什么走下一步
- **等价动作**: 为了让别人复现同类观察, 补充的最小命令集合

等价动作的作用是帮助复盘和培训, 不是把后补命令伪装成原始执行历史.

## 事后已知的故障制作方式

用户事后披露, 故障是这样制造的:

```bash
printf '\xff%.0s' {1..100} | dd of=journal/WiredTigerLog.0000000007 bs=3 count=2096 conv=notrunc
```

然后重启故障 Pod.

这条信息只适合放在案例包里做特征对照. 它不应该出现在现场首诊的起点位置.
